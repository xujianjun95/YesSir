## 🔴 P0(立刻修)

### 1. 遥测数据**静默丢失** — `bg-telemetry.js`
- `sendTelemetry` 用 try/catch 把网络/HTTP 错误全部吞掉,`flushPendingDailyCounters` 紧接着 `delete counters[date]`,与注释"失败下次再试"完全相反。
- `fetchWithTimeout` 也不检查 `response.ok`,服务端 5xx 也被视为成功 → 累计点击数据丢得无声。
- **改**:`sendTelemetry` 失败抛错,flush 仅在全部 feature 成功后才 delete。

### 2. 日期口径不一致 — `bg-messages.js:110` vs `bg-telemetry.js:26`
- `recordUsage` 用 **UTC**(`toISOString().slice(0,10)`),遥测用**本地**(`getLocalDateKey()`)。
- 东八区凌晨 0~8 点:`dailyStats` 已跳次日,`ysDailyCounters` 仍在前日 → 看板/激活率全错位。
- **改**:统一调用 `getLocalDateKey()`。

### 3. 修饰键"双击"误触发 — `content.js:1123`
- 判定双击时只看 `event.key === 'Meta'`,**不排除组合键**。用户连按 Cmd+S、Cmd+R、Cmd+T 时,两次 keydown 的 key 都是 'Meta',300ms 内连按必中 → 面板意外弹出。
- **改**:`!event.altKey && !event.shiftKey && !event.ctrlKey`,只接受纯修饰键单按。

### 4. keydown capture 阶段无差别拦截 — `content.js:1089–1160`
- Mod+E、双击修饰键 keydown 在 capture 阶段 `preventDefault + stopPropagation`,不检查 target 是否为 `input/textarea/[contenteditable]`。
- 后果:Notion、Figma、VSCode Web、富文本编辑器里的快捷键被扩展吞掉。
- **改**:开头 `if (event.target.closest('input,textarea,[contenteditable]')) return;`

### 5. content 侧 `sendMessage` 无 try/catch — `content.js:300/325/957`
- 扩展更新/禁用后,旧 content script 仍存活但 `chrome.runtime.id` 失效 → **同步抛** `Extension context invalidated`。`ysRuntimeSendMessageRetry` 本身就在第 300 行抛,重试根本接不住。
- visibilitychange ping 一直挂着,每次切回页面就再抛一次。
- **改**:包 try/catch,catch 里 `cb(null,'invalidated')` 并自卸 visibilitychange 监听。

### 6. 拖拽 mouseup 不触发 → 幽灵缩略图 — `content-switcher/04-show-switcher.js:1318/1798`
- Esc / Cmd+Tab 切窗口 / 拖出浏览器外松开 → `mouseup` 不触发,`floatEl` + document mousemove 全部残留。
- **改**:mousedown 时一并注册 `window.blur` + `keydown(Esc)` 兜底清理。

### 7. ResizeObserver 泄漏 — `content-switcher/04-show-switcher.js:2068`
- 每次 `showSwitcher` 新建一个 RO 挂在被 remove 的 `card` 上,从不 disconnect → 开关 N 次堆 N 个孤儿 RO。
- **改**:`hideSwitcher` 里 `pinnedColRO.disconnect()`。

### 8. AI 快照缓存随 tabId 复用残留 — `bg-ai-network.js`
- `aiSnapshotCache.entries` 用 `tab.id` 作 key,**`chrome.tabs.onRemoved` 中没有同步清理**。Chrome 关闭后 tab id 会被回收复用,旧 entry 一直留到 600 条上限才裁。
- **改**:在 `bg-core.js` 的 `onRemoved` 里 `delete aiSnapshotCache.entries[String(tabId)]`。

### 9. proxy-server 限流形同虚设 — `proxy-server/server.js`
- `usageMap` 纯内存:**重启清零、pm2 多 worker 各自计数**。
- `getTodayStr()` 用 **UTC**:北京时间早上 8 点用户配额突然刷新。
- `Access-Control-Allow-Origin: *`:任意网页都能直接调用消耗 UUID 配额。
- **改**:落 Redis/SQLite + 改 `Asia/Shanghai` + CORS 仅放行 `chrome-extension://<id>`。

## 推荐修复顺序

1. **#2 日期口径**(1 行改动,影响数据可信度)
2. **#3 修饰键误判**(高频用户痛点,Cmd+S 必中)
3. **#4 keydown 拦截**(影响第三方编辑器)

5. **#5 sendMessage 异常**(扩展更新后控制台爆红)
6. **#6/#7 拖拽 + RO**(内存/幽灵 bug)
7. **#8 aiSnapshotCache**(长期内存)
8. **#9 proxy 限流**(后端工作)

## 总体结论

**架构没毛病**,核心问题集中在三处:

1. **遥测/计数层**日期口径与失败处理不一致(数据可信度受损)
2. **content 侧 keydown 状态机**在组合键 + 富文本场景下容易误触发或抢按键
3. **资源生命周期**(ResizeObserver、mouseup 兜底、aiSnapshotCache、proxy 限流)的清理路径不闭合
